# card 테이블은 다음과 같은 방식으로 설계하였습니다.

- card_items (공통 카드 규칙)
- card_item_texts, card_item_videos, card_item_images 

# 이 설계에 장점과 단점은 다음과 같습니다.


## 장점

1. 확장성이 좋다. 

  - 새로운 타입에 카드를 만들기 쉽다 ( 공통 카드 테이블 + 도형 테이블 등등 )
  
  - 각 타입에 테이블에 열을 추가하기 쉽다 ( 각 테이블 열에 새로운 타입을 추가하기가 편합니다. )

2. 관리하기가 좋습니다.

  - 개발자가 개발을 할때 비교적 쉽습니다.

  - 하나의 테이블씩 관리가 가능합니다.

## 단점

1. 성능 저하 문제

  - join을 여러번 해야 하다보니 성능적인 문제가 발생할 수 있습니다.

2. 개발에서 신경써주어야 할 부분이 많다. 

  - 저장할때 정합성이라던지 그러한 부분 등 신경써야 할 부분이 늘어날수 있습니다.

## 선택이유

  - 추가 적인 기능이 생길 가능성이 농후 하다 

  - 수정 가능성이 농후 하다. 

## 단점 보안을 위한 노력

  - redis를 이용한 캐싱 전략 활용 

  - cards와 card_items는 fk 관계를 갖지만 card_items와 각 요소는 fk 관계를 갖지 앉고 인덱싱을 이용해서 관리해준다. 

  - union과 각 card_id는 안다는 가정하에 1번의 join을 이용하는 방식으로 성능을 높인다.
